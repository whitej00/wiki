# NAT (Network Address Translation)

RFC 2663, RFC 3022에 정의된 네트워크 주소 변환 기술. 사설 네트워크 내부의 **사설 IP 주소**를 단일 (또는 소수의) **공인 IP 주소**로 변환하여 외부 인터넷과 통신할 수 있게 한다.

## 동작 원리

NAT 라우터는 외부에서 보면 **단일 IP 주소를 가진 하나의 장치**처럼 보인다.

### 나가는 패킷 (LAN → WAN)
1. 내부 호스트(10.0.0.1:3345)가 외부 서버(128.119.40.186:80)에 패킷 전송
2. NAT 라우터가 출발지 IP를 공인 IP(138.76.29.7)로 교체
3. 출발지 포트를 새 포트(5001)로 교체
4. **NAT 변환 테이블**에 매핑 기록: (138.76.29.7, 5001) ↔ (10.0.0.1, 3345)
5. 변환된 패킷을 외부로 전송

### 들어오는 패킷 (WAN → LAN)
1. 외부 서버의 응답이 (138.76.29.7, 5001)로 도착
2. NAT 라우터가 변환 테이블을 조회
3. 목적지를 (10.0.0.1, 3345)로 복원하여 내부로 전달

## NAT 변환 테이블

| WAN 측 | LAN 측 |
|--------|--------|
| 138.76.29.7, 5001 | 10.0.0.1, 3345 |
| ... | ... |

포트 번호가 16비트이므로, 단일 공인 IP로 **~60,000개** 동시 연결을 지원할 수 있다.

## 사설 주소 공간

RFC 1918이 정의한 사설 네트워크용 주소 범위:
- **10.0.0.0/8** (10.0.0.0 ~ 10.255.255.255)
- **172.16.0.0/12** (172.16.0.0 ~ 172.31.255.255)
- **192.168.0.0/16** (192.168.0.0 ~ 192.168.255.255)

이 주소들은 사설 네트워크 내부에서만 의미를 가지며, 인터넷에서 라우팅되지 않는다.

## 장점

- **주소 절약**: 수백~수천 개의 내부 장치가 하나의 공인 IP 공유
- **보안**: 내부 네트워크 구조가 외부에 노출되지 않음
- **유연성**: ISP 변경 시 내부 주소 재설정 불필요

## 비판과 문제점

1. **포트 번호 오용**: 포트는 프로세스 식별용이지 호스트 식별용이 아님
2. **계층 원칙 위반**: 라우터(L3)가 포트 번호(L4)를 수정
3. **서버 운영 어려움**: NAT 뒤의 호스트는 외부에서 직접 접근 불가
4. **P2P 문제**: 두 호스트가 모두 NAT 뒤에 있으면 직접 연결 곤란

## NAT Traversal

NAT 뒤의 서버에 접근하기 위한 기술:
- **STUN** (RFC 5389): 외부에서 보이는 공인 IP:포트를 확인
- **TURN** (RFC 5128): 중계 서버 사용
- **UPnP**: NAT 라우터에 포트 매핑을 자동 요청
