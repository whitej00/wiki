# 미들박스 (Middleboxes)

RFC 3234 정의: "소스 호스트와 목적지 호스트 사이의 데이터 경로에서, 표준 IP 라우터 기능 외의 기능을 수행하는 중간 장치."

## 3가지 유형

### 1. NAT 변환
[[NAT]] 장치가 사설 네트워크 주소를 공인 주소로 변환. IP 주소와 포트 번호를 재작성한다.

### 2. 보안 서비스
- **방화벽(Firewall)**: 헤더 필드 기반으로 의심스러운 데이터그램/세그먼트를 차단. TCP 연결 추적, 소스/목적지 IP·포트 기반 필터링
- **IDS(Intrusion Detection System)**: **심층 패킷 검사(DPI)** — 헤더뿐 아니라 페이로드(애플리케이션 데이터)까지 검사. 알려진 공격 시그니처와 매칭
- **IPS(Intrusion Prevention System)**: IDS + 탐지된 패킷을 실제로 차단

### 3. 성능 향상
- **압축**: 데이터 압축으로 대역폭 절약
- **콘텐츠 캐싱**: 웹 캐시 등
- **로드 밸런싱**: 서비스 요청을 여러 서버에 분산

## NFV (Network Function Virtualization)

SDN의 접근 방식을 미들박스에 적용:
- 별도의 전용 하드웨어 대신, **범용 서버 위의 소프트웨어**로 미들박스 기능 구현
- 과거: 별도 NAT 장비, 별도 방화벽 장비, 별도 로드밸런서 → 각각 다른 소프트웨어 스택, 관리 부담
- NFV: 범용 하드웨어 + 공통 소프트웨어 스택 위에 가상화된 네트워크 기능 배포

## IP 모래시계 (IP Hourglass)

인터넷 프로토콜 스택은 **모래시계 형태**이다:
- 상위 (넓음): HTTP, SMTP, QUIC, DASH 등 다양한 애플리케이션 프로토콜
- 중간 (좁음): **IP 단 하나** — 모든 장치가 반드시 구현해야 하는 유일한 프로토콜
- 하위 (넓음): Ethernet, WiFi, Bluetooth, 광섬유 등 다양한 링크 기술

IP의 단순함과 유일성이 다양한 상·하위 기술을 연결하는 인터넷 성장의 핵심이었다.

그러나 미들박스의 증가로 이 "좁은 허리"가 넓어지고 있다. [[NAT]]는 IP 주소·포트를 재작성하고, 방화벽은 애플리케이션 계층까지 검사하며, 계층 간의 깔끔한 분리가 흐려지고 있다.

## End-to-End 논거와 미들박스

RFC 1958의 핵심 원칙: *"지능은 네트워크 코어가 아닌 종단에 배치"* [Saltzer 1984].

미들박스는 이 원칙에 도전한다:
- 과거: 네트워크 코어는 포워딩만 수행, 복잡한 기능은 종단 호스트에서 처리
- 현재: NAT, 방화벽, DPI 등이 네트워크 내부에서 상위 계층 정보를 처리

미들박스를 "아키텍처적 혐오"로 보는 시각과, "중요한 필요를 충족하므로 불가피"로 보는 시각이 공존한다.
