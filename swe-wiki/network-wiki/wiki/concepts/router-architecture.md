# 라우터 아키텍처 (Router Architecture)

라우터는 네트워크 계층의 핵심 장비로, 입력 링크에서 출력 링크로 데이터그램을 포워딩하는 **패킷 스위치**이다.

## 4대 구성요소

### 1. 입력 포트 (Input Port)
- **라인 종단**: 물리 계층 — 물리 링크의 신호 수신
- **링크 계층 처리**: 링크 계층 프로토콜 처리 (역캡슐화)
- **룩업/포워딩**: 포워딩 테이블을 조회하여 출력 포트 결정 → 스위칭 패브릭으로 전달
- 포워딩 테이블은 라우팅 프로세서에서 **섀도 복사본**을 받아 로컬에서 조회 → 중앙 병목 방지

### 2. 스위칭 패브릭 (Switching Fabric)
입력 포트에서 출력 포트로 패킷을 실제 전달하는 라우터 내부 네트워크.

| 방식 | 특징 | 성능 |
|------|------|------|
| **메모리** | 라우팅 프로세서가 메모리에 복사. 공유 시스템 버스 사용 | 최대 B/2 (B=메모리 대역폭) |
| **버스** | 입력→출력을 공유 버스로 직접 전달. 내부 라벨로 출력 포트 식별 | 한 번에 하나만. 소규모 충분 |
| **교차바(Crossbar) 네트워크** | 2N 버스, N×N 교차점. **Non-blocking** — 다른 출력이면 병렬 전달 가능 | 고성능. 다단계로 확장 가능 |

### 3. 출력 포트 (Output Port)
- 스위칭 패브릭에서 받은 패킷을 **큐잉(버퍼 관리)** → 링크 계층 처리 → 라인 종단(전송)
- [패킷 스케줄링](packet-scheduling.md) 수행

### 4. 라우팅 프로세서 (Routing Processor)
- **제어 플레인** 기능 수행: 라우팅 프로토콜 실행, 라우팅 테이블 유지, 포워딩 테이블 계산
- SDN 라우터에서는 원격 컨트롤러와 통신하여 포워딩 테이블 수신
- 네트워크 관리 기능도 수행
- 소프트웨어, 밀리초/초 단위

## 큐잉과 패킷 손실

### 입력 큐잉
스위칭 패브릭이 입력 라인 속도를 따라가지 못하면 입력 포트에서 큐잉 발생.

**HOL(Head-of-Line) 블로킹**: 큐 맨 앞의 패킷이 출력 포트 경합으로 차단되면, 뒤에 있는 패킷도 (출력 포트가 비어있음에도) 대기해야 하는 현상. 입력 링크 이용률 58%에서 이미 큐가 무한히 증가할 수 있다.

### 출력 큐잉
여러 입력에서 동시에 같은 출력 포트로 패킷이 도착하면 출력 큐 형성. 버퍼가 가득 차면 **패킷 손실** 발생.

폐기 정책:
- **Drop-tail**: 도착 패킷을 폐기
- **기존 패킷 제거**: 버퍼가 차기 전에 선제적으로 폐기/마킹
- **AQM(Active Queue Management)**: RED, PIE, CoDel 등

### 버퍼 크기

- 경험적 규칙: B = RTT · C (소수 TCP 흐름)
- 다수의 독립 TCP 흐름(N개): **B = RTT · C / √N**
- 10 Gbps 링크, RTT=250ms → B = 2.5 Gbit (소수) 또는 훨씬 적음 (다수)

### Bufferbloat

과도한 버퍼로 인해 큐잉 지연이 **지속적이고 과도하게** 발생하는 현상. 처리량은 유지되지만 지연이 극심. 게이머나 화상통화 사용자에게 치명적. DOCSIS 3.1이 AQM 도입으로 대응.

## 하드웨어 vs 소프트웨어

- 데이터 플레인 (입력/출력 포트, 스위칭 패브릭): **하드웨어** — 100 Gbps 링크에서 64바이트 패킷은 5.12ns 내 처리 필요
- 제어 플레인 (라우팅 프로세서): **소프트웨어** — CPU
