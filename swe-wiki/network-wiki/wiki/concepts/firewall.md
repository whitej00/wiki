# 방화벽 (Firewall)

조직의 내부 네트워크와 공용 인터넷 사이에서 트래픽을 검사하여 허용/차단하는 하드웨어+소프트웨어 조합.

## 방화벽의 세 가지 목표

1. **모든 트래픽이 방화벽을 통과**해야 함 (단일 출입구)
2. **보안 정책에 따라 허가된 트래픽만 통과** 허용
3. **방화벽 자체가 침투에 면역**이어야 함

## 방화벽 유형

### 1. 전통적 패킷 필터 (Traditional Packet Filter)

게이트웨이 라우터에서 각 데이터그램을 **개별적으로** 검사하여 통과/차단을 결정한다.

**필터링 기준**:
- IP 소스/목적지 주소
- 프로토콜 타입 (TCP, UDP, ICMP, OSPF 등)
- TCP/UDP 소스/목적지 포트
- TCP 플래그 비트 (SYN, ACK 등)
- ICMP 메시지 타입
- 인바운드/아웃바운드 방향

**ACL (Access Control List)**: 라우터의 각 인터페이스에 적용되는 규칙 목록. 위에서 아래로 매칭하며 최초 매치 규칙 적용.

예시 정책:

| 정책 | 방화벽 설정 |
|------|-----------|
| 외부 웹 접근 차단 | 포트 80 대상 아웃바운드 패킷 차단 |
| 웹 서버만 인바운드 TCP 허용 | 130.207.244.203:80 외 인바운드 SYN 차단 |
| 인터넷 라디오 차단 | 인바운드 UDP 패킷 차단 (DNS 제외) |
| Smurf DoS 방지 | 브로드캐스트 주소 대상 ICMP ping 차단 |
| Traceroute 방지 | 아웃바운드 ICMP TTL expired 차단 |

**ACK 비트 트릭**: 외부→내부 TCP 연결 차단 시, ACK=0인 패킷(= SYN, 즉 연결 시작)만 차단하면 내부→외부 연결은 허용하면서 외부 시작 연결만 차단 가능.

### 2. 상태 기반 패킷 필터 (Stateful Packet Filter)

전통적 필터의 한계: ACK=1 + 소스 포트 80인 외부 패킷을 허용해야 하는데, 악의적 패킷도 이 조건을 만족할 수 있음.

**해결**: **연결 테이블(Connection Table)**을 유지하여 진행 중인 TCP 연결을 추적.

| 소스 주소 | 목적지 주소 | 소스 포트 | 목적지 포트 |
|----------|----------|----------|----------|
| 222.22.1.7 | 37.96.87.123 | 12699 | 80 |

- 3-way handshake(SYN, SYNACK, ACK) 관찰로 연결 시작 감지
- FIN 또는 60초 무활동으로 연결 종료 판단
- ACL에 "check connection" 열 추가 → 연결 테이블에 없는 외부 ACK 패킷 차단

### 3. 애플리케이션 게이트웨이 (Application Gateway)

IP/TCP/UDP 헤더를 넘어 **애플리케이션 데이터** 수준에서 정책을 적용한다.

- 애플리케이션별 전용 서버 (Telnet 게이트웨이, HTTP 프록시 등)
- 사용자 ID/비밀번호로 **사용자 인증** 가능
- 모든 트래픽이 게이트웨이를 경유하도록 패킷 필터와 결합
- 단점: 애플리케이션별로 별도 게이트웨이 필요, 성능 오버헤드, 클라이언트 설정 필요

## 익명성과 프라이버시

방화벽과 관련된 주제로, 프록시 + SSL 또는 **TOR**(The Onion Router)를 사용하여:
- 웹사이트에 IP 주소를 숨기고
- ISP에 방문 사이트를 숨기고
- 교환 데이터를 암호화

TOR는 3개 프록시 체인을 랜덤 선택하여 라우팅하며, 프록시들이 결탁하지 않는 한 누가 누구와 통신하는지 알 수 없다.
