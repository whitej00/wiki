# 신뢰적 데이터 전송 (Reliable Data Transfer)

네트워킹에서 가장 근본적인 문제 중 하나. 하위 채널이 비신뢰적(비트 오류, 패킷 손실)이더라도, 상위 계층에 **데이터가 손상 없이, 중복 없이, 순서대로** 전달되는 것처럼 보이는 서비스를 구현하는 것이다.

## 왜 필요한지

IP 서비스는 **비신뢰적(unreliable)**이다. 데이터그램이 손실되거나, 순서가 바뀌거나, 비트가 손상될 수 있다. 신뢰적 데이터 전송은 전송 계층뿐 아니라 링크 계층, 애플리케이션 계층에서도 필요한 범용 문제이다.

## 서비스 모델

- **송신 측**: `rdt_send(data)` 호출 → 신뢰적 전송 프로토콜이 `udt_send(packet)`으로 비신뢰 채널에 전달
- **수신 측**: `rdt_rcv(packet)` 이벤트 → 신뢰적 전송 프로토콜이 `deliver_data(data)`로 상위 계층에 전달

## 프로토콜 단계적 구축

### rdt1.0: 완전 신뢰 채널

하위 채널이 완전히 신뢰적인 경우. 오류 처리가 전혀 불필요하다.
- 송신: 데이터 → 패킷 생성 → 전송
- 수신: 패킷 수신 → 데이터 추출 → 상위 전달

### rdt2.0: 비트 오류 채널 — ARQ 프로토콜

비트 오류가 발생할 수 있는 채널. **ARQ(Automatic Repeat reQuest)** 프로토콜의 세 가지 요소:
1. **오류 검출**: 체크섬으로 비트 오류 감지
2. **수신자 피드백**: **ACK**(긍정 확인응답)와 **NAK**(부정 확인응답) 전송
3. **재전송**: NAK 수신 시 패킷 재전송

rdt2.0은 **stop-and-wait** 프로토콜: 송신자가 ACK/NAK를 받을 때까지 다음 데이터를 보내지 않음.

**치명적 결함**: ACK/NAK 자체가 손상될 수 있음!

### rdt2.1: 손상된 ACK/NAK 처리 — 순서번호 도입

ACK/NAK가 손상되면 송신자는 현재 패킷을 재전송한다. 이때 수신자가 **중복 패킷을 구별**할 수 있도록 **순서번호(Sequence Number)**를 도입한다.

stop-and-wait 프로토콜에서는 **1비트 순서번호(0과 1)**로 충분하다.

### rdt2.2: NAK-free 프로토콜

NAK 대신 **마지막으로 정상 수신한 패킷의 순서번호를 포함한 ACK**를 전송한다. 송신자가 **중복 ACK(duplicate ACK)**를 받으면 NAK와 동일하게 처리한다.

이 방식은 이후 TCP의 fast retransmit 메커니즘의 기초가 된다.

### rdt3.0: 패킷 손실 채널 — 타이머 도입

비트 오류에 더해 **패킷 손실**이 발생할 수 있는 채널. 데이터 패킷이나 ACK가 손실되면 송신자에게 아무 응답이 도착하지 않는다.

해결: **카운트다운 타이머(Countdown Timer)**를 도입. 적절한 시간 내에 ACK가 도착하지 않으면 재전송한다.

- 조기 타임아웃(premature timeout)으로 중복 패킷이 발생할 수 있으나, 순서번호로 처리 가능
- 순서번호가 0과 1을 교대하므로 **교대 비트 프로토콜(Alternating-bit Protocol)**이라고도 함

## 파이프라이닝 (Pipelining)

rdt3.0(stop-and-wait)의 문제: **채널 이용률이 극히 낮다**.

예시 계산:
- 1 Gbps 링크, RTT = 30ms, 패킷 크기 = 1,000바이트
- 전송 시간 d_trans = 8,000비트 / 10^9 bps = 8μs
- 이용률 U_sender = (L/R) / (RTT + L/R) = 0.008 / 30.008 = **0.00027 (0.027%)**

해결: **ACK를 기다리지 않고 여러 패킷을 연속 전송**하는 파이프라이닝. 3개 패킷을 파이프라이닝하면 이용률이 3배 증가.

파이프라이닝의 요구사항:
- 순서번호 범위 증가 (각 패킷에 고유 순서번호 필요)
- 송신자/수신자 버퍼링 필요
- 오류 복구 방식에 따라 **Go-Back-N**과 **Selective Repeat** 두 접근

## Go-Back-N (GBN)

**슬라이딩 윈도우 프로토콜(Sliding-window Protocol)**이라고도 한다.

### 송신자

- 윈도우 크기 **N**: 미확인 패킷의 최대 허용 수
- 순서번호 공간: `[base, base+N-1]`이 송신 가능 범위
- **누적 확인응답(Cumulative ACK)**: ACK(n)은 n 이하 모든 패킷의 정상 수신을 의미
- 단일 타이머: 가장 오래된 미확인 패킷에 대해. 타임아웃 시 **윈도우 내 전체 미확인 패킷 재전송**

### 수신자

- **매우 단순**: 기대 순서번호의 패킷만 수용, 나머지는 **폐기**
- 비순서 패킷을 버퍼링하지 않음
- 항상 가장 최근 정상 수신 패킷의 ACK를 재전송

### 장단점
- 장점: 수신자 구현 단순 (버퍼링 불필요)
- 단점: 단일 패킷 오류로 윈도우 전체를 재전송해야 하므로 **대역폭 낭비** 가능

## Selective Repeat (SR)

GBN의 불필요한 재전송 문제를 해결한다. 손실/오류가 의심되는 패킷**만** 재전송한다.

### 송신자

- 윈도우 크기 N
- **개별 확인응답(Individual ACK)**: 각 패킷에 대해 독립적 ACK
- **패킷별 논리 타이머**: 각 미확인 패킷에 개별 타이머 (실제로는 단일 하드웨어 타이머로 시뮬레이션)
- ACK 수신 시 해당 패킷을 수신 완료로 표시. `send_base` 패킷의 ACK이면 윈도우 전진

### 수신자

- 윈도우 범위 내의 정상 패킷은 **버퍼링** (비순서여도)
- 각 패킷에 대해 **개별 ACK** 전송
- `rcv_base` 패킷 수신 시, 버퍼링된 연속 패킷들을 한꺼번에 상위 계층에 전달
- 윈도우 아래쪽(`[rcv_base-N, rcv_base-1]`)의 패킷도 재확인응답 (송신자 윈도우 전진을 위해)

### 윈도우 크기 제한

송신자와 수신자의 윈도우가 항상 동기화되지 않으므로, 유한한 순서번호 공간에서 혼동을 방지하려면:

**윈도우 크기 ≤ 순서번호 공간 크기 / 2**

## 신뢰적 데이터 전송 메커니즘 요약

| 메커니즘 | 용도 |
|----------|------|
| 체크섬 | 비트 오류 검출 |
| 타이머 | 손실 패킷 타임아웃/재전송 |
| 순서번호 | 패킷 순서 추적, 중복 감지 |
| 확인응답(ACK) | 정상 수신 알림 (개별 또는 누적) |
| 부정 확인응답(NAK) | 오류 수신 알림 |
| 윈도우/파이프라이닝 | 채널 이용률 향상을 위한 다중 패킷 전송 |

## 패킷 재정렬 문제

패킷이 네트워크 내에서 재정렬될 수 있다. 이전 연결의 순서번호가 재사용되면 혼동 가능. 실무에서는 패킷이 네트워크에 약 **3분**(최대 패킷 수명) 이상 존재하지 않는다고 가정하여, 순서번호를 충분한 시간이 지난 후에만 재사용한다.

## TCP에서의 적용

[[TCP]]는 GBN과 SR의 **하이브리드**이다:
- GBN적: 누적 확인응답, 가장 작은 미확인 바이트(SendBase) 추적
- SR적: 비순서 세그먼트 버퍼링, 손실 세그먼트만 재전송
- Selective Acknowledgment(SACK, RFC 2018) 확장으로 비순서 수신 정보를 명시적으로 전달 가능
