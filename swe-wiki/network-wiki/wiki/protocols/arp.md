# ARP (Address Resolution Protocol)

**IP 주소를 MAC 주소로 변환**하는 프로토콜 (RFC 826). 링크 계층과 네트워크 계층의 경계에 위치하며, 같은 서브넷 내의 호스트/라우터 인터페이스에 대해서만 동작한다.

## 왜 필요한가

프레임을 전송하려면 목적지의 [[mac-address|MAC 주소]]가 필요하다. 하지만 상위 계층은 IP 주소만 알고 있다. ARP는 이 간극을 메우는 역할을 한다.

> DNS가 호스트 이름 → IP 주소를 인터넷 전역에서 해석하는 반면, ARP는 IP 주소 → MAC 주소를 **같은 서브넷 내에서만** 해석한다.

## 동작 원리

### ARP 테이블

각 호스트와 라우터는 **ARP 테이블**을 메모리에 유지한다:

| IP 주소 | MAC 주소 | TTL |
|---------|---------|-----|
| 222.222.222.221 | 88-B2-2F-54-1A-0F | 13:45:00 |
| 222.222.222.223 | 5C-66-AB-90-75-B1 | 13:52:00 |

- 테이블은 서브넷 내 모든 노드를 반드시 포함하지 않을 수 있음
- 각 엔트리에는 **TTL** 존재 (일반적으로 20분, 만료 시 삭제)

### ARP 질의/응답

1. 송신 호스트가 목적지 IP에 대한 MAC 주소를 ARP 테이블에서 찾지 못함
2. **ARP 질의 패킷(ARP Query)**을 생성하여 MAC 브로드캐스트 주소(`FF-FF-FF-FF-FF-FF`)로 전송
3. 서브넷의 **모든 어댑터**가 ARP 질의를 수신하고 자신의 ARP 모듈로 전달
4. 목적지 IP가 일치하는 노드만 **ARP 응답 패킷(ARP Reply)**을 유니캐스트로 송신 호스트에게 전송
5. 송신 호스트가 ARP 테이블을 업데이트

> **ARP 질의는 브로드캐스트**, **ARP 응답은 유니캐스트**

### Plug-and-Play

ARP 테이블은 **자동으로 구축**된다. 관리자 설정이 필요 없으며, 호스트가 서브넷에서 연결 해제되면 엔트리가 자연스럽게 만료된다.

## 서브넷 간 전송에서의 ARP

호스트 A(서브넷 1)가 호스트 B(서브넷 2)에게 데이터그램을 보낼 때:

1. A는 IP 데이터그램의 목적지 IP를 B로 설정
2. 프레임의 목적지 MAC 주소는 B가 아닌 **첫 번째 홉 라우터의 인터페이스 MAC 주소**로 설정 (ARP로 획득)
3. 라우터가 프레임을 수신 → IP 데이터그램 추출 → 포워딩 테이블 참조 → 서브넷 2의 인터페이스로 전달
4. 라우터가 B의 MAC 주소를 ARP로 획득하여 새 프레임 생성 후 전송

> IP 데이터그램의 출발지/목적지 IP는 불변이지만, **프레임의 MAC 주소는 각 홉에서 변경**된다.

## ARP의 계층 위치

ARP는 분류가 애매한 프로토콜이다:
- ARP 패킷은 링크 계층 프레임 안에 캡슐화됨 → 링크 계층?
- ARP 패킷에 IP 주소 등 네트워크 계층 정보 포함 → 네트워크 계층?

결론적으로 ARP는 **링크 계층과 네트워크 계층의 경계에 걸친 프로토콜**이다.
