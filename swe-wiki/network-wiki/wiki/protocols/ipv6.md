# IPv6 (Internet Protocol version 6)

RFC 2460, RFC 4291에 정의된 차세대 인터넷 프로토콜. **128비트 주소**로 IPv4의 주소 고갈 문제를 해결하고, 헤더를 간소화하여 라우터 처리 속도를 향상시킨다.

## 등장 배경

1990년대 초 IPv4의 32비트 주소 공간(약 43억 개)이 고갈될 것으로 예측. 2011년 IANA가 마지막 IPv4 블록을 지역 레지스트리에 할당. IPv6는 이 문제를 근본적으로 해결하기 위해 개발되었다.

+                                                               +

+                                                               +
|                                                               |

+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+-+

### 주요 필드

- **Version** (4비트): 6
- **Traffic Class** (8비트): IPv4의 TOS와 동일 기능. 우선순위 구분, ECN 포함
- **Flow Label** (20비트): 특정 플로우에 속하는 패킷 식별. QoS 처리에 활용
- **Payload Length** (16비트): 40바이트 헤더 이후의 페이로드 크기
- **Next Header** (8비트): 페이로드의 프로토콜. IPv4의 Protocol 필드와 동일 값 사용. 옵션 헤더 체이닝에도 사용
- **Hop Limit** (8비트): IPv4의 TTL과 동일. 매 홉마다 1 감소, 0이면 폐기
- **Source/Destination Address** (각 128비트): 주소 형식은 RFC 4291

헤더 크기: **40바이트 고정** (IPv4의 가변 길이와 대조)

## IPv4 대비 주요 변경

### 추가된 것
- **128비트 주소**: 2^128 ≈ 3.4×10^38개. "지구의 모래알 하나하나에 IP 할당 가능"
- **Anycast 주소**: 그룹 내 가장 가까운 호스트에 전달 (예: 미러 서버 중 최근접 선택)
- **Flow Label**: 플로우별 특별 처리 요청 가능

### 제거된 것
- **단편화/재조립**: 중간 라우터에서의 단편화 금지. MTU 초과 시 "Packet Too Big" ICMP 오류 → 송신자가 크기 조정. 라우터 처리 속도 향상
- **헤더 체크섬**: 전송/링크 계층(TCP, UDP, 이더넷)에서 이미 체크섬 수행하므로 중복. IPv4에서는 TTL 변경 때문에 매 홉마다 재계산해야 했음 → 큰 비용 절감
- **Options 필드**: 표준 헤더에서 제거. Next Header를 통해 확장 헤더(extension header)로 체이닝하여 유연하게 처리

## IPv4에서 IPv6로의 전환: 터널링

IPv4와 IPv6는 호환되지 않는다. 전체 인터넷을 한 번에 전환하는 것(flag day)은 불가능.

**터널링(Tunneling)**: IPv6 데이터그램 전체를 IPv4 데이터그램의 **페이로드**에 캡슐화하여 IPv4 라우터를 통과시키는 방식.

동작:
1. IPv6 노드 B가 IPv6 데이터그램을 IPv4 데이터그램에 캡슐화 (protocol 번호 = 41)
2. 중간 IPv4 라우터들은 일반 IPv4 패킷으로 처리
3. 터널 출구의 IPv6 노드 E가 IPv4 페이로드에서 IPv6 데이터그램 추출

## 채택 현황

- 초기에는 느렸으나 가속 중
- Google 서비스의 ~25% 클라이언트가 IPv6 사용
- 미국 정부 2차 도메인의 1/3 이상이 IPv6 지원
- 3GPP가 모바일 멀티미디어에 IPv6를 표준 주소 체계로 지정
- 네트워크 계층 프로토콜 교체는 "집의 기초를 바꾸는 것"에 비유될 만큼 어려움
